## BUG-001: POST /api/1/item принимает отрицательный price

**Краткое описание:** API создаёт объявление с `price=-100`, хотя цена не может быть отрицательной.

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом:
   ```json
   {"sellerID": 555123, "name": "Товар с отрицательной ценой", "price": -100, "statistics": {"likes": 1, "viewCount": 1, "contacts": 1}}
   ```

**Фактический результат:** 200 OK, объявление создано. Цена `price=-100` сохраняется в базе.

**Ожидаемый результат:** 400 Bad Request - цена не может быть отрицательной.

**Серьёзность:** Major

**Автотест:** `TestCreateItemNegativePrice`

---

## BUG-002: POST /api/1/item отклоняет нулевые значения в поле statistics

**Краткое описание:** При передаче `{"likes": 0, "viewCount": 0, "contacts": 0}` API возвращает 400 "поле likes обязательно". Нулевые значения должны быть допустимы (отсутствие лайков - нормальное состояние нового объявления).

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом:
   ```json
   {"sellerID": 555234, "name": "Товар", "price": 100, "statistics": {"likes": 0, "viewCount": 0, "contacts": 0}}
   ```

**Фактический результат:** 400 Bad Request: `{"result": {"message": "поле likes обязательно", ...}, "status": "400"}`.

**Ожидаемый результат:** 200 OK - объявление создано. Нулевые значения счётчиков корректны.

**Серьёзность:** Major

**Автотест:** `TestCreateItemZeroStatisticsBug`

---

## BUG-003: POST /api/1/item зависает при sellerID=-1

**Краткое описание:** При передаче отрицательного `sellerID=-1` сервер не возвращает ответ - соединение зависает (timeout).

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом:
   ```json
   {"sellerID": -1, "name": "Товар", "price": 100, "statistics": {"likes": 1, "viewCount": 1, "contacts": 1}}
   ```

**Фактический результат:** Соединение зависает (timeout клиента).

**Ожидаемый результат:** 400 Bad Request немедленно.

**Серьёзность:** Critical

**Автотест:** `TestCreateItemSellerIDMinusOneHangOrTimeout` - POST с `sellerID=-1` и ограничением времени ожидания клиента; при зависании тест завершается по таймауту (успех сценария «зафиксирован дефект»), после исправления API ожидается быстрый ответ **400**.

## BUG-004: POST /api/1/item принимает отрицательные значения statistics

**Краткое описание:** API создаёт объявление с отрицательными счётчиками (`likes=-1`, `viewCount=-1`, `contacts=-1`), хотя статистика не может быть отрицательной.

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом:
   ```json
   {"sellerID": 555123, "name": "Товар", "price": 100, "statistics": {"likes": -1, "viewCount": 1, "contacts": 1}}
   ```

**Фактический результат:** 200 OK, объявление создано с отрицательным значением likes=-1.

**Ожидаемый результат:** 400 Bad Request - счётчики не могут быть отрицательными.

**Серьёзность:** Major

**Автотест:** `TestCreateItemNegativeStatisticsLikes`, `TestCreateItemNegativeStatisticsViewCount`, `TestCreateItemNegativeStatisticsContacts`

## BUG-006: GET /api/1/:sellerID/item с отрицательным sellerID возвращает 200

**Краткое описание:** Запрос объявлений продавца с отрицательным `sellerID` (например, `-111111`) возвращает 200 OK вместо 400.

**Шаги воспроизведения:**
1. Отправить `GET /api/1/-111111/item`

**Фактический результат:** 200 OK, пустой массив или список объявлений.

**Ожидаемый результат:** 400 Bad Request - отрицательный sellerID недопустим.

**Серьёзность:** Minor

**Автотест:** `TestGetSellerItemsNegativeSellerID`

## BUG-007: POST /api/1/item принимает price=2^53+1

**Краткое описание:** API создаёт объявление с `price=9007199254740993` (2^53+1). Такое значение выходит за пределы целых, точно представимых в IEEE-754 double (как в JSON); ожидается отказ валидации.

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом:
   ```json
   {"sellerID": 555123, "name": "Очень дорогой товар", "price": 9007199254740993, "statistics": {"likes": 1, "viewCount": 1, "contacts": 1}}
   ```

**Фактический результат:** 200 OK, объявление создано (цена может исказиться при разборе JSON).

**Ожидаемый результат:** 400 Bad Request - цена вне допустимого диапазона точных целых.

**Серьёзность:** Minor

**Автотест:** `TestCreateItemVeryLargePrice`

## BUG-008: POST /api/1/item принимает отрицательный sellerID (-111111)

**Краткое описание:** API создаёт объявление с `sellerID=-111111`, хотя идентификатор продавца не должен быть отрицательным. (Значение `-1` не проверять автотестом - зависание, BUG-003.)

   ```json
   {"sellerID": -111111, "name": "Тест отрицательный продавец", "price": 1000, "statistics": {"likes": 1, "viewCount": 1, "contacts": 1}}
   ```

**Фактический результат:** 200 OK, объявление создано.

**Ожидаемый результат:** 400 Bad Request - отрицательный sellerID недопустим.

**Серьёзность:** Minor

**Автотест:** `TestCreateItemNegativeSellerIDPost`

**См. также:** BUG-006 (тот же класс проблемы для GET списка по sellerID).

## BUG-009: POST /api/1/item принимает чрезмерно длинный name (100 000 символов)

**Краткое описание:** API создаёт объявление с `name` длиной 100 000 символов и отвечает **200 OK**, хотя такая длина должна отклоняться валидацией.

**Шаги воспроизведения:**
1. Отправить `POST /api/1/item` с телом, где поле `name` - строка из 100 000 символов, остальные поля валидны (в т.ч. `statistics` с ненулевыми счётчиками).

**Фактический результат:** 200 OK, объявление создано.

**Ожидаемый результат:** 400 Bad Request - превышена допустимая длина названия.

**Серьёзность:** Minor

**Автотест:** `TestCreateItemNameTooLong`

## BUG-010: GET /api/1/item/:id - поле `createdAt` с дублирующимся смещением часового пояса

**Краткое описание:** В ответе `GET /api/1/item/:id` (и в элементах списка `GET /api/1/:sellerID/item`) поле `createdAt` иногда приходит в виде строки с **двойным** хвостовым смещением, например `2026-04-04 20:22:05.822885 +0300 +0300`. Такая строка **не соответствует** типичному ISO 8601 / RFC3339 и **не парсится** стандартным `time.Parse` с одним смещением без предобработки.

**Шаги воспроизведения:**
1. Создать объявление через `POST /api/1/item`.
2. Выполнить `GET /api/1/item/{id}`.
3. Просмотреть значение поля `createdAt` в JSON.

**Фактический результат:** Строка даты может заканчиваться на повтор смещения (`+HHMM +HHMM` или аналог с `-`).

**Ожидаемый результат:** Одна временная зона в конце строки (как в RFC3339), без дубликата, например `2026-04-04T20:22:05.822885+03:00` или эквивалент с одним смещением.

**Серьёзность:** Minor

**Автотест:** `TestGetItemCreatedAtFormat` - парсинг **сырой** строки `createdAt` из ответа; при дубле смещения тест падает.

**См. также:** `API_DISCREPANCIES.md` (раздел про `createdAt`).
