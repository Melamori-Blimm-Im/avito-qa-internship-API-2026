# Расхождения реального API с Postman-коллекцией

При написании и запуске автотестов оказалось, что реальное API `https://qa-internship.avito.com` работает не так, как описано в Postman-коллекции - расхождений оказалось немало.

## Сводная таблица расхождений

| #   | Эндпоинт         | Что задокументировано                          | Реальное поведение                                                     | Тип |
| --- | ---------------- | ---------------------------------------------- | ---------------------------------------------------------------------- | --- |
| 1   | POST /api/1/item | Success response содержит полный объект с `id` | Response содержит только `{"status": "Сохранили объявление - <UUID>"}` |
| 2   | Все ошибки       | `result.messages` - объект с ключами           | `result.messages` - всегда пустой `{}`                                 |
| 3   | GET /api/1/item/:id (и объекты в списке продавца) | `createdAt` как типичная дата в духе ISO 8601 / RFC3339 | Строка с пробелом вместо `T` и **дублирующимся** смещением, напр. `2026-04-04 20:22:05.822885 +0300 +0300` |


---

## 1. POST /api/1/item - формат тела ответа при успехе

### Postman-коллекция

Ответ `200 OK` описан как полный объект объявления:

```json
{
  "id": " ",
  "sellerId": " ",
  "name": " ",
  "price": " ",
  "statistics": {
    "likes": " ",
    "viewCount": " ",
    "contacts": " "
  },
  "createdAt": " "
}
```

### Реальное поведение

Ответ `200 OK` содержит только строку со статусом и UUID:

```json
{
  "status": "Сохранили объявление - c6f1cc30-5694-4758-a090-6a258d1e2206"
}
```

**Влияние:** UUID созданного объявления нельзя получить из поля `id` - его нужно парсить из строки `status` по паттерну `"Сохранили объявление - <UUID>"`.

## 2. Формат ответа при ошибке - структура отличается

### Postman-коллекция

Ответ `400 Bad Request` описан как:

```json
{
  "result": {
    "messages": {
      "culpa_b92": " ",
      "enim_24f": " ",
      "mollit_aa": " "
    },
    "message": " "
  },
  "status": " "
}
```

### Реальное поведение

Реальная структура ошибки:

```json
{
  "result": {
    "message": "поле likes обязательно",
    "messages": {}
  },
  "status": "400"
}
```

Отличия:
- `status` - строка, содержащая HTTP-код (`"400"`), а не произвольная строка
- `result.messages` - всегда пустой объект `{}` при ошибках валидации (поле `messages` в коллекции показано как непустой объект с ключами)
- Информативное сообщение находится в `result.message`, а не в ключах `result.messages`

## 3. Поле `createdAt` в успешных ответах GET (объявление)

### Ожидаемое / задокументированное поведение

В коллекции и в типичных REST-контрактах `createdAt` обычно задаётся в  формате ISO 8601 или RFC3339 (одна временная зона, без повторов), например `2026-04-04T20:22:05Z` или `2026-04-04T20:22:05+03:00`.

### Реальное поведение

В ответе `GET /api/1/item/:id` (и в элементах списка `GET /api/1/:sellerID/item`) поле `createdAt` приходит как строка вида:

```text
2026-04-04 20:22:05.822885 +0300 +0300
```

### Формат даты и времени

Дата и время разделены **пробелом** вместо буквы `T`, и часовой пояс дублируется `+0300 +0300` - парсер ломается. 
**Что это значит на практике:** прежде чем валидировать дату, придётся привести строку к нормальному виду - например, убрать лишний суффикс. Иначе проверка на «корректную дату» будет давать ложные срабатывания.
